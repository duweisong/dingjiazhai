# DESIGN 维护 SOP

> 元规则:规定 `DESIGN.md` 该写什么、不该写什么、什么时候写。
> 与 `文档维护SOP.md`(管 `CLAUDE.md`)**平行、解耦**。
> **仅当项目有设计 / 视觉 / 创意工作时启用**;纯调研 / 内容 / 逻辑类项目可不建 `DESIGN.md`。

---

## 1. 写什么
- **设计 / 创意方向决策 + 为什么**:为什么这么定、否定了哪些替代方案
- **外部设计 / 视觉参考**:链接、范例、风格参考(带日期与用途)
- **设计系统 / 风格规范的来源**:配色、字号、间距、组件或物料规范的出处
- **重大设计历史节点**:大改版、方向调整(带绝对日期,如 `2026-03`)

## 2. 不写什么
- 工程 / 方案 / 技术决策 → 归 `CLAUDE.md`
- 产物本身(成品、具体实现就是)
- 一次性的评审意见

## 3. 与 `CLAUDE.md` 的边界
| 内容 | 归属 |
|---|---|
| 为什么这么**做**(方案 / 结构 / 技术) | `CLAUDE.md` |
| 为什么这么**呈现**(设计 / 视觉 / 创意) | `DESIGN.md` |
| 整体布局、模块协议、跨模块约定 | `CLAUDE.md` |
| 视觉规范、设计参考、风格来源 | `DESIGN.md` |

## 4. 触发时机
出现以下信号时,按「提议 → 确认 → 写入」主动提议更新 `DESIGN.md`:
- 新增 / 替换 / 更新设计 / 视觉参考
- 设计 / 交互 / 创意方向决策(含否定的替代方案)
- 设计系统 / 方向的重大调整

## 5. 操作与同步
- 操作流程同 `文档维护SOP.md` §5:识别 → 提议(列修改点)→ 确认 → 写入 → 保持精简。
- 根级 `CLAUDE.md` 在引用索引里指一句向 `DESIGN.md`。
- `DESIGN.md` 与 `CLAUDE.md` 相互引用处,两侧同步更新。

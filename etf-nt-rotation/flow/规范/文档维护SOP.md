# 文档维护 SOP

> 元规则:规定 `CLAUDE.md` 体系该写什么、不该写什么、谁在什么时候写、写到哪一层、怎么写。
> 目的:让 Claude Code 和 Codex 按同一套标准维护文档,避免漂移、膨胀或遗漏。
> **适用于任何项目(代码 / 调研 / 内容 / 方案),不绑定技术栈。**

---

## 1. 写入标准(什么该进文档)
只沉淀**非显而易见、看产物本身看不出来**的内容:
- **设计哲学 / 心智模型**:把这件事理解成什么(例:"任务型而非聊天型")
- **关键决策 + 背后的"为什么"**:不是"我们用了 X",而是"为什么选 X、否定了 Y"
- **方向定位**:目标对象、核心命题、一句话定位
- **环节 / 模块间的契约**:交接口径、字段约定、依赖关系
- **废弃 / 禁区**:哪些东西不要动、哪些做法不要再用
- **外部参考资料**:链接、调研、竞品、范例(带日期和用途)
- **非显而易见的约定**:违背常规习惯的项目特例
- **重要历史节点**:重大调整、废弃决策(带绝对日期,如 `2026-03`)

## 2. 不写标准(什么不进文档)
- **看产物本身就知道的**:文件清单、目录结构、内容本身
- **版本历史能查到的**(git log / blame 等):谁改的、改了啥
- **一次性状态**:当前 TODO、临时进度
- **通用常识 / 最佳实践**(除非项目有特例)
- **个别问题的修复细节**(修复本身在产物和提交记录里)

## 3. 触发时机(什么时候写)
出现以下信号时**主动提议**写入(不擅自写,列修改点等确认):
- 用陈述句描述了一条新的**设计哲学 / 心智模型 / 方向定位** → `CLAUDE.md`
- 本轮做了**结构 / 方案调整**(新增模块、改契约、废弃某部分)→ `CLAUDE.md`
- 分享了**需要持续参考的外部资料** → `CLAUDE.md`(根级引用索引)
- 出现"**以后都这么办 / 不要再 X / 记住 Y**"的长期约定 → `CLAUDE.md`
- 每轮结束前主动按以上核对。hook 只是一层保险,没 hook 的会话也要自觉过一遍。
- (若项目有设计 / 创意工作,另有一条 → `DESIGN.md`,见 §7)

## 4. 分层归属(写到哪一份)
### 4.1 建立原则
**每个有独立职责的子模块 / 子项目都建一份 `CLAUDE.md`**(不是按需才建)。纯工具 / 杂项目录无需单建。
### 4.2 各层职责
| 层级 | 写什么 |
|---|---|
| **根级 `/CLAUDE.md`** | (1) 跨模块的哲学、定位、整体布局、通用约定;(2) 从踩坑提炼出的**禁区 / 约定**(详细问题日志在 `flow/踩坑记录.md`);(3) 对外部资料、子模块文档、可选 `DESIGN.md` 的引用索引 |
| **子模块 `<module>/CLAUDE.md`** | 仅限本模块的心智模型、内部约定、模块内契约、模块级踩坑——**不跨边界写别的模块** |
原则:**就近写**,能写子模块就不上根级,避免根级膨胀;跨模块内容才上根级。

## 5. 操作流程(怎么写)
1. **识别**:出现触发信号
2. **归属**:判断进哪一层
3. **提议**:列出具体修改点(加在哪节、几行),等确认
4. **写入**:确认后用 Edit 追加 / 修改;只改 `CLAUDE.md`(`AGENTS.md` 是软链,自动同步)
5. **保持精简**:能一行不写三行;非显而易见才留;过期内容顺手删

## 6. 同步更新(跨层变更时)
一次变更跨多层时(如改跨模块契约、心智模型演进),**相关各层都要同步更新**——只改一处会让另一处迅速过期。
- 改根级契约 → 同步涉及的子模块文档
- 改子模块心智 / 废弃策略 → 若影响跨模块理解,根级补一条
- 发现某层过期,顺手修或删
- 若用了 `DESIGN.md`,它与 `CLAUDE.md` 相互引用处两侧同步
原则:**文档同步跟着变更走**,一轮里改完就顺手把文档改完,不拖到下一轮。

## 7. 设计 / 创意类内容 → 见 `DESIGN.md`(单独成文)
设计、视觉、创意方向的决策**不进 `CLAUDE.md`**,归 `DESIGN.md`;其写入规范在**独立的 `DESIGN维护SOP.md`**(仅当项目有设计 / 创意工作时启用)。
边界:"为什么这么**做**(方案 / 结构)"→ `CLAUDE.md`;"为什么这么**呈现**(设计 / 创意)"→ `DESIGN.md`。

## 反例速查
- ❌ "今天修了一个小问题" → 修复细节,不写
- ❌ "某文件里有某字段" → 看产物就知道,不写
- ✅ "X 模块是 A 心智而非 B,2026-03 从旧方案迁移" → 心智 + 历史,写根级
- ✅ "环节间约定新增某口径用于某用途" → 跨模块契约,写根级

## 8. 文档自检 Hook
每轮结束的自检由 Stop hook 兜底。**配置与说明见 `hook机制.md`**,此处不重复。

## 9. 原则
1. **产物本身是真相的唯一来源**(文档只补"看产物看不出来的为什么")。
2. **解决问题从根本上解,不打补丁。**
